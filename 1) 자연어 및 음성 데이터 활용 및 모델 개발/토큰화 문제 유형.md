#  "토큰화(tokenization)에서 발생하는 경계 결정 문제”를 유형별로 정리

---

# 1. 토큰화의 본질적 문제

토큰화는 단순히 “공백 기준 분리”가 아니라
👉 **언어적 의미를 유지하면서 최소 단위로 나누는 문제**입니다.

즉, 핵심 질문은 항상 동일합니다:

> “이 표현을 하나의 의미 단위로 볼 것인가, 여러 개로 분해할 것인가?”

---

# 2. 주요 토큰화 이슈 (구조화 정리)

## (1) 구두점 처리 (Punctuation Handling)

### 예

* `Hello, world!`

### 선택지

* 분리: `["Hello", ",", "world", "!"]`
* 미분리: `["Hello,", "world!"]`

### 판단 기준

* **문법 정보가 중요한 경우** → 분리 (NLP 모델)
* **단순 검색/카운팅** → 미분리 가능

### 영향

* 분리하면 **문장 구조 분석에 유리**
* 미분리하면 **데이터 단순화**

---

## (2) 축약형 처리 (Contraction Handling)

### 예

* `"don't"` → `["do", "n't"]`
* `"we're"` → `["we", "'re"]`

### 선택지

* 분해 (linguistic split)
* 유지 (surface form 유지)

### 판단 기준

* **언어적 의미 분석 (POS, parsing)** → 분해
* **문장 자연성 유지 (generation)** → 유지

### 영향

* 분해 시:

  * `"n't"` → 부정 의미 명확
* 유지 시:

  * 모델이 패턴을 통째로 학습

---

## (3) 합성어 처리 (Multi-word Expression)

### 예

* `"New York"`
* `"ice cream"`

### 선택지

* 분리: `["New", "York"]`
* 결합: `["New_York"]` 또는 `"New York"` 하나로 처리

### 판단 기준

* **고유명사 / 관용 표현** → 하나로 유지
* 일반 조합 → 분리

### 영향

* 결합 시:

  * 의미 보존 ↑ (특히 NER, embedding)
* 분리 시:

  * 일반화 능력 ↑

---

## (4) 하이픈 처리 (Hyphenated Words)

### 예

* `"state-of-the-art"`

### 선택지

* 완전 분리: `["state", "of", "the", "art"]`
* 부분 분리: `["state-of-the-art"]`
* 하이브리드: `["state-of-the-art", "state", "art"]`

### 판단 기준

* **의미가 하나로 고정된 경우** → 유지
* **조합 가능한 일반 단어** → 분리

### 영향

* 유지:

  * 의미 정확성 ↑
* 분리:

  * 데이터 희소성 감소

---

# 3. 핵심 판단 기준 (통합 관점)

토큰화 결정은 아래 3가지 축으로 판단합니다:

### ① 의미 보존 (Semantic Integrity)

* 의미 단위가 유지되는가?

### ② 모델 목적 (Task Dependency)

* 분석(Parsing, NER) vs 생성(LLM, 번역)

### ③ 데이터 효율성 (Statistical Efficiency)

* vocabulary 크기 vs 일반화 성능

---

# 4. 현대 NLP에서의 해결 방식

전통적인 rule-based 토큰화 대신:

### Subword Tokenization (BPE, WordPiece 등)

* `"don't"` → `["don", "'t"]`
* `"state-of-the-art"` → 더 작은 단위로 분해

👉 장점:

* OOV 문제 해결
* 희귀 단어 처리 가능

---

# 5. spaCy 관점에서 보면

spaCy는 기본적으로:

* 구두점 분리 ✔
* 축약형 분리 ✔ (`n't`, `'re`)
* 고유명사는 유지하려고 시도 ✔
* 하이픈은 규칙 기반 처리 ✔

즉,
👉 **언어학적 일관성을 우선하는 tokenizer**

> “의미, 문법, 모델 목적 사이의 균형을 잡는 설계 문제”
